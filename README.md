# SHROOM — Telegram-бот знакомств

aiogram 3 · SQLAlchemy 2 (async) · PostgreSQL · APScheduler · Telegram Stars

## Запуск

```bash
pip install -r requirements.txt
cp .env.example .env      # заполнить BOT_TOKEN, DATABASE_URL, FIRST_ADMIN_ID
python bot/main.py
```

Таблицы и недостающие колонки создаются автоматически при старте.
Для эволюции схемы в проде рекомендуется перейти на alembic (он уже в requirements).

## Реферальная система

Каждый пользователь получает диплинк `https://t.me/<bot>?start=ref_<его_id>`
(кнопка «🔗 пригласить друзей» в профиле или команда `/invite`).

- Новичок регистрируется по ссылке → **обоим** начисляются бонусные свайпы
  (`REFERRAL_WELCOME_SWIPES` новичку, `REFERRAL_BONUS_SWIPES` рефереру).
- Бонусные свайпы (`users.bonus_swipes`) суммируются с базовым `SWIPE_LIMIT`.
- Реферер получает уведомление о каждом приглашённом.
- Защита от накруток: нельзя пригласить себя; привязка реферера ставится
  один раз; начисление только после реально завершённой регистрации.

Параметры — в `bot/constants.py` (`REFERRAL_*`).

## Антиспам (исправлено)

Раньше троттлинг (`CommandThrottleMiddleware`) реагировал только на команды
(`/...`), а в боте почти всё взаимодействие — нажатия инлайн-кнопок и кнопок
меню. Поэтому лимит фактически ни на что не влиял.

Теперь ограничиваются:
- нажатия инлайн-кнопок (`callback_query`) — `CALLBACK_RATE` (0.6 с) — это
  гасит «дребезг» двойных тапов и флуд лайк/диз/навигация;
- команды — `COMMAND_RATE` (2 с).

Свободный текст не троттлится намеренно — иначе ломались бы пошаговые
сценарии (регистрация, редактирование, рассылка). Лимиты — в
`bot/middlewares/throttle.py`.

## SHROOM+ (премиум-подписка)

Платная подписка через Telegram Stars (на той же инфраструктуре, что и платное
удаление анкеты). Кнопка «✦ shroom+» в профиле или команда `/premium`.

Покупка — разовый платёж Stars. Несколько тарифов в `bot/constants.py`
(`PREMIUM_PLANS` — список `(дней, звёзд)`): на витрине каждый тариф это кнопка,
payload вида `premium_v2:<days>`, срок берётся из payload при оплате. Повторная
покупка продлевает «стопкой» (от текущей даты окончания).

Что входит:
- 🩸 **повышенный лимит свайпов** — `PREMIUM_SWIPE_LIMIT` (по умолчанию 500) вместо базовых 50, плюс реферальные бонусы;
- 🔝 **приоритет в ленте** — премиум-анкеты сортируются выше (не ломая гео-приоритет);
- ↩️ **откат свайпа** — команда `/undo` удаляет последний свайп (и снимает мэтч, если тот образовался), анкета снова появляется в ленте;
- ✦ **бейдж** в карточке анкеты.

Срок хранится в `users.premium_until` (TIMESTAMPTZ), колонка добавляется
автоматически при старте. Число активных подписок видно в админ-статистике.

Технически: и удаление анкеты, и премиум ловят `successful_payment`, поэтому оба
хэндлера фильтруются по `invoice_payload` — иначе первый сработавший остановил бы
распространение апдейта и второй платёж не обработался бы.

### Идеи для развития SHROOM+ (не реализовано)
- фильтры поиска (возраст/пол/расстояние) только для премиума;
- супер-лайк с мгновенным уведомлением получателя;
- режим «инкогнито» (смотреть анкеты, не оставляя следов до взаимности);
- нативная авто-продлеваемая подписка Stars (`subscription_period`) вместо разовых покупок;
- порог для реферального бонуса (начислять, когда приглашённый сделал N свайпов) — защита от фарма.

## Антиспам (доработано)

Троттлинг (`bot/middlewares/throttle.py`) теперь ограничивает три вида действий,
у каждого свой независимый кулдаун:
- инлайн-кнопки (`callback_query`: лайк/диз/навигация) — 0.6 с;
- кнопки нижнего меню (reply-keyboard «🕯️ лента» и т.п.) — 1.0 с
  (это текстовые сообщения, ловятся по списку `MENU_BUTTON_TEXTS` в constants);
- команды (`/...`) — 2.0 с.

Произвольный текст не троттлится, чтобы не ломать пошаговые сценарии.

Регистрируется одним общим экземпляром как `outer_middleware` на
`dp.message` и `dp.callback_query` — это гарантирует вызов для каждого
сообщения и нажатия кнопки. Команды блокируются при ЧАСТОЙ отправке
(чаще, чем раз в `COMMAND_RATE`=2с); вручную в это окно обычно не попасть,
поэтому при медленном наборе блок незаметен — значение настраивается.

## Рендер карточек

Формирование карточек в `bot/utils/formatting.py` разделено на чистые
функции `render_profile_caption` / `render_own_profile` (без БД) и
async-обёртки, которые подтягивают статистику. Чистые функции легко
рендерятся для превью без подключения к базе.

## Лайк с сообщением

В ленте у каждой анкеты есть кнопка «💬 лайк+» — лайк вместе с короткой
запиской (до `LIKE_MSG_MAX_LEN`=200 символов). Лимит — `LIKE_MSG_LIMIT_PER_HOUR`=5
записок в час на пользователя (скользящее окно, см. constants).

Поток: тап «💬 лайк+» → ввод текста (есть кнопка «✖️ отмена») → лайк
регистрируется как обычно (может сразу дать мэтч), записка доставляется
получателю отдельным сообщением с карточкой отправителя и кнопками
«🩸 в ответ» / «✖️ пропустить», а также показывается в разделе «🩸 лайки». Если ответить взаимностью — образуется мэтч.

Записки хранятся в таблице `like_messages` (создаётся автоматически через
`create_all`). Вход в пункты нижнего меню сбрасывает FSM — это заодно
страхует от «застрявших» сценариев ввода.

## Доработки лайка с сообщением

- Лимит снижен до 5 записок в час.
- Исправлено «не отображается»: получателю передавался «лёгкий» объект
  пользователя без загруженных фото, и обращение к `.photos` падало (async
  lazy-load), а сообщение глоталось `except`. Теперь во все уведомления
  (мэтч, доставка записки) передаётся полностью загруженный объект с фото.
  Тот же фикс применён к обычным уведомлениям о мэтче.
- Записка показывается и в разделе «🩸 лайки» (latest_from).

## Клавиатуры

- В ленте лайк (🩸) слева, дизлайк (🤮) справа.
- Меню профиля структурировано в ряды: редактирование (1) · скрыть/уведы (2)
  · shroom+/пригласить (2) · удалить (1) — вместо длинного столбца из 6 кнопок.

## SHROOM+ — лимит вместо безлимита

Премиум даёт `PREMIUM_SWIPE_LIMIT` (по умолчанию 500) вместо безлимита —
это «гораздо больше», но не бесконечно. Реферальные бонусы суммируются сверху.

## Бейдж премиума

Значок премиума в карточке вынесен в `PREMIUM_BADGE` (constants), по умолчанию
💎 — цветной эмодзи в тон остальным. В титулах/кнопках SHROOM+ остаётся
фирменный типографский ✦.

## Фильтры поиска

Кнопка «🔍 фильтры» в профиле и команда `/filters`.

- **🎂 возраст** (всем): диапазон через дефис (`18-25`), `0` — сброс. Поля
  `users.age_min` / `age_max`, фильтр `User.age BETWEEN ...` в `get_next_candidate`.
- **📍 расстояние** (для всех): пресеты 5/10/25/50/100/200 км или без
  ограничения. Жёсткий гео-фильтр (`users.max_distance_km`), применяется если
  у юзера указана геолокация.
- Пол (`looking_for`) остаётся в «редактировать», в фильтрах показан справочно.

Колонки добавляются автоматически миграцией (`ADD COLUMN IF NOT EXISTS`).
Сброс — кнопка «♻️ сбросить».

## Кто смотрел анкету

Просмотры пишутся в таблицу `profile_views` (одна строка на пару
viewer→target, время обновляется при повторном показе). Фиксируются при
показе карточки в ленте (`_send_card`).

Раздел «👀 кто смотрел» (кнопка в профиле + команда `/views`):
- бесплатно — только счётчик: «твою анкету смотрели: N»;
- в SHROOM+ — постранично сами анкеты смотревших (свежие первыми, с «N назад»),
  с реакцией 🩸/🤮 и репортом.

Это вернуло SHROOM+ премиум-ценность после того, как фильтр по расстоянию
стал доступен всем.

## Админка: тестовая выдача SHROOM+

В админ-меню — кнопка «🔱 выдать shroom+» (ввод telegram ID) и кнопка
«🔱 +30д shroom+» прямо на карточке найденного юзера. Выдаёт `PREMIUM_DAYS`
дней без оплаты — для тестов. Под капотом тот же `UserRepository.grant_premium`
(продлевает стопкой, если премиум уже активен).

## Тарифы SHROOM+ и возврат к анкете

- **Несколько сроков покупки.** `PREMIUM_PLANS = [(дней, звёзд), ...]` (по умолчанию
  7/30/90/365 дней). Витрина показывает каждый тариф кнопкой; оплата идёт по
  payload `premium_v2:<days>`, при `successful_payment` срок парсится из payload.
  Хэндлеры premium фильтруются по префиксу payload (delete-анкеты — по равенству),
  чтобы апдейты платежей не перехватывали друг друга.
- **Админ-выдача с выбором срока.** Кнопка «🔱 выдать shroom+» (ввод ID) и
  «🔱 +shroom+» на карточке юзера → выбор срока (те же пресеты из `PREMIUM_PLANS`) →
  `grant_premium(days)`. Без оплаты, для тестов.
- **Возврат к анкете — кнопка вместо команды.** Команда `/undo` убрана. В ленте под
  карточкой кнопка «↩️ вернуть прошлую анкету» (SHROOM+): откатывает последний свайп
  (снимает мэтч, если образовался; пересчитывает рейтинг) и сразу показывает
  возвращённую анкету. Не-премиуму — подсказка про /premium.

## Тестирование без живых тестеров

Главная проблема дейтинг-бота: почти всё (мэтчи, лайки, записки, просмотры,
репорты) требует второго пользователя. Подходы:

1. **Тест-режим в админке (основной).** Кнопка «🧪 тест-режим» в админ-меню.
   Генерит синтетические анкеты (`id < 0` — не пересекаются с реальными Telegram
   id) и симулирует взаимодействия:
   - «🧪 +10 анкет» — наполняет ленту (свайпы, фильтры, приоритет премиума, рендер);
   - «🩸 5 лайкнут меня» — входящие лайки → раздел «🩸 лайки», мэтч по ответу;
   - «💬 записка мне» — лайк с запиской в «лайках»;
   - «👀 7 просмотров» — раздел «кто смотрел» (нужен SHROOM+);
   - «⚔️ мэтч с фейком» — раздел «мэтчи»;
   - «🚩 создать репорт» — очередь модерации в «репорты»;
   - «🧹 удалить тестовые» — `DELETE WHERE id<0`, связанные лайки/мэтчи/просмотры/
     записки/репорты уносятся каскадом по FK.
   Фейкам можно слать уведомления — они молча падают (обёрнуты в try/except).
   Чтобы видеть «кто смотрел/лайкнул», выдай себе SHROOM+ (там же в админке).

2. **Второй аккаунт** — для финальной проверки живого UX (оплата Stars, диплинки).

3. **Скриптовые проверки** — репозитории/сервисы и чистые функции рендера
   тестируются без Telegram (как делалось для троттлинга и карточек).

## Возврат к анкете — только последняя из ЛЕНТЫ и в рамках захода

Кнопка «↩️ вернуть прошлую анкету» откатывает строго последний свайп из ЛЕНТЫ
(реакции из «лайки»/«кто смотрел» не считаются) и только В ТЕКУЩЕМ ЗАХОДЕ.
Указатель `rewind_target` хранится в сессионном FSM-состоянии: ставится в
`handle_like`/`handle_dislike`, сбрасывается при входе в любой пункт меню
(лента/лайки/мэтчи) и при перезапуске бота. Вернуть можно ровно одну, самую
недавнюю анкету; после возврата указатель обнуляется. В начале захода или если
фид-свайпов ещё не было — «нечего возвращать — это первая анкета в заходе».

## Кто смотрел мой профиль (видимый счётчик)

Просмотры пишутся в `profile_views` при показе карточки в ленте. В своём профиле
теперь видно «👀 N просмотров», а кнопка «👀 кто смотрел» (и /views) открывает
раздел: бесплатно — счётчик, в SHROOM+ — кто именно (постранично, со временем
«N назад» и реакцией). Для проверки в одиночку используй админ-тест-режим:
«👀 7 просмотров».

## «Кто смотрел» — живой счётчик

Раздел и счётчик исключают только тех, кого Я УЖЕ ОТРЕАГИРОВАЛ (лайк/диз — из
раздела или из ленты): после моей реакции человек уходит из списка и счётчика
(фильтр по `Like.from_user==me`). Все остальные смотревшие остаются, включая
тех, кто меня дизлайкнул.

Фильтрация — в самом запросе (`view_repo._pending_cond`), поэтому счётчик в
профиле и список в разделе всегда совпадают. Сам факт просмотра по-прежнему
пишется при показе карточки в ленте.

Счётчик просмотров встроен в строку статистики профиля: `🩸 · 🤮 · ⚔️ · 👀`
(вместо отдельной строки).

## Компактная клавиатура ленты

Кнопки под карточкой: `🩸 🤮 ↩️` в одном ряду + `💬 лайк+ 🚩` — два ряда
компактных кнопок вместо широкой кнопки возврата во всю строку. Это уменьшает
скачки интерфейса на мобильном при смене карточек.

## Город + расстояние

Карточка теперь показывает «🏙 город · N км». Город определяется обратным
геокодированием (Nominatim/OpenStreetMap) ОДИН раз — при сохранении локации
(регистрация и «редактировать → геолокация»), и хранится в `users.city`.
На каждый показ карточки запрос не делается. Геокодер вызывается с таймаутом
и мягким фолбэком: при ошибке/таймауте город просто не ставится (остаётся
только расстояние). aiohttp импортируется лениво. Свой профиль показывает
«🏙 город» вместо «гео не указана», если город известен.

Учти политику Nominatim (≤1 запрос/сек, нужен User-Agent) — для боевой
нагрузки лучше платный геокодер; здесь запрос редкий (только при смене гео).

## Тест-режим: «🩸 ответит на лайк»

Кнопка делает мэтч с тем фейком, кого ты только что лайкнул в ленте: берётся
твой последний лайк на фейк без мэтча, фейк «лайкает в ответ», создаётся мэтч.
Если лайков на фейков нет — подсказка «сначала лайкни фейка». Фейкам при
генерации теперь проставляется город (видно фичу «город · км»).

## Тест-режим: исправления

- **Уведомления фейкам не шлются.** Функции `_send_match_notification`,
  `_notify_liked`, `_deliver_like_message` пропускают получателей с `id < 0`
  (у фейков нет чата). Это убирает шум «Bad Request: chat not found» в логах —
  раньше эти строки выглядели как «сломанные функции», хотя падали ожидаемо.
- **Тест-действия берут «свежих» фейков.** «лайкнут меня / записка / просмотры /
  мэтч / репорт» теперь нацеливаются на фейков, которых админ ЕЩЁ НЕ свайпал
  (если все пролайканы — досоздаются новые). Иначе результат прятался фильтром
  «уже отреагировал», и казалось, что кнопки не работают.

## Умный подбор в ленте

`get_next_candidate` теперь ранжирует кандидатов по сигналам (в порядке важности):
1. **кто меня уже лайкнул** — выше всех (максимальный шанс мэтча; коррелированный
   `EXISTS` по входящим лайкам);
2. **ближние** (гео-bbox), если у меня указана локация;
3. **взаимный интерес** — кандидат ищет мой пол или «всех»;
4. **SHROOM+** приоритет;
5. ближе по дистанции;
6. рандом (для разнообразия).

Базовые фильтры (пол/возраст/дистанция/уже-свайпнутые/активные) — как были.

## Лента без «скачков» (edit вместо delete+send)

Свайп (лайк/диз) и возврат теперь **редактируют то же сообщение-карточку**
(`edit_message_media`) вместо «удалить + прислать заново» — мобильный интерфейс
больше не подпрыгивает. Конец ленты гасит карточку через `edit_message_caption`.
Если редактировать нельзя (карточка без фото / несовместимый тип) — мягкий
фолбэк на старое поведение delete+send. Первая карточка захода и доставка из
других разделов отправляются как раньше.
